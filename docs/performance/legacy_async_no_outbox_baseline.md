# 旧版全异步链路压测记录（无 Outbox）

## 概述

本文档记录的是项目在引入 Outbox Pattern 之前，旧版全异步抽奖链路的一次历史基线压测结果。

这组数据的作用是用于体现轻链路、理想化条件下的性能上限，不应直接视为当前生产型架构下的最终性能结果。

## 测试条件

- 机器配置：单机 `2C4G`
- 压测工具：`wrk`
- 测试用户数：`5000`
- 链路版本：`无 Outbox`
- 用户额度：所有用户均提前配置了充足额度
- 额度预热：测试前已将用户活动额度提前加载到 Redis
- 奖品库存：库存充足，提前完成准备
- 测试目的：评估缓存预热场景下，抽奖核心链路的吞吐与延迟基线

## 核心指标

| 指标 | 数据 |
| --- | --- |
| QPS（吞吐量） | `6002 req/s` |
| Avg Latency（平均延迟） | `49.75 ms` |
| P50 Latency（50% 用户） | `49.39 ms` |
| P90 Latency（90% 用户） | `56.38 ms` |
| P99 Latency（99% 用户） | `62.62 ms` |
| Total Requests（总请求数） | `60145` |
| Error Rate（错误率） | `0%` |
| CPU：MySQL（`mysqld`） | `< 2%` |
| CPU：Go 应用（`app_main`） | `~165%` |

- 在无 Outbox 的基线版本下，系统在单机 `2C4G` 环境中可达到约 `6000 QPS`。
- 延迟分布整体较稳定，`P99` 约为 `62 ms`，没有出现明显的长尾抖动。
- 错误率为 `0%`，说明在该测试条件下，核心链路运行稳定。
- MySQL CPU 占用低于 `2%`，说明这次压测中数据库并不是主要瓶颈。
- Go 应用进程 CPU 约为 `165%`，说明系统压力更集中在应用层与缓存访问路径。

## 适用范围与限制

- 这组结果属于引入 Outbox 之前的性能基线测试。
- 由于额度与库存都已经提前准备并完成预热，因此这组数据更接近轻链路下的性能上限。
- 在引入 Outbox、本地消息表写入、异步投递、失败补偿等机制后，系统吞吐通常会有所下降，这是为了换取更强一致性与更高可靠性的正常工程取舍。
