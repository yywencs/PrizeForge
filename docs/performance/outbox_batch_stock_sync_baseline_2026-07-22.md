# Outbox 批量库存同步压测基线（v1.0.5）

## 概述

本文档记录 PrizeForge `v1.0.5` 在单机 `2C4G` 环境中的完整抽奖链路压测结果。

该版本将 `strategy_award_stock_sync` Outbox 任务按照
`(strategy_id, award_id)` 分组，同组任务在一个事务中登记订单幂等凭证，并将多次奖品库存
扣减合并为一次数据库更新。测试目的是验证该优化对热点库存行锁竞争、接口吞吐和延迟的影响。

优化前的完整原始指标、资源快照、Outbox 积压和慢 SQL 证据见
[逐任务库存同步压测基线（v1.0.3～v1.0.4）](./pre_batch_stock_sync_baseline_2026-07-21.md)。

## Outbox 状态快照

并发 40 测试结束后查询两个分库的 `task` 表：

| 分库 | Topic | 状态 | 数量 |
| --- | --- | --- | ---: |
| `prizeforge_01` | `send_award` | `completed` | `16109` |
| `prizeforge_01` | `send_award` | `create` | `10939` |
| `prizeforge_01` | `strategy_award_stock_sync` | `completed` | `15901` |
| `prizeforge_01` | `strategy_award_stock_sync` | `create` | `11102` |
| `prizeforge_02` | `send_award` | `completed` | `15897` |
| `prizeforge_02` | `send_award` | `create` | `11109` |
| `prizeforge_02` | `strategy_award_stock_sync` | `completed` | `15898` |
| `prizeforge_02` | `strategy_award_stock_sync` | `create` | `11108` |

当时共有 `44258` 条 `create` 状态任务。`prepare` 不会清理历史订单、中奖记录和任务，因此这份快照可能包含
此前压测遗留任务，不能将其全部视为本轮新增任务。

当前扫描器每 5 秒扫描一次，每个分库一轮最多读取 500 条任务。两个分库的理论扫描上限约为：

```text
2 × 500 ÷ 5s = 200 tasks/s
```

每次成功抽奖会产生 `send_award` 和 `strategy_award_stock_sync` 两条 Outbox。并发 25 时的输入速度约为：

```text
112.37 requests/s × 2 = 224.74 tasks/s
```

该速度已经超过扫描器的理论读取上限；如果某轮处理超过 5 秒，进程内互斥锁还会跳过重叠调度，实际消费能力会更低。

## 慢 SQL 观察

本次提供的慢 SQL 样本主要包括：

1. 按 `user_id + order_id + draw_state` 抢占抽奖订单执行权；
2. 按 `user_id + activity_id + request_id` 查询幂等订单；
3. 按 `user_id + message_id` 逐条更新 Outbox 状态。

样本中没有再次出现批量改造前 `strategy_award` 库存行扣减的慢 SQL，说明同一策略奖品批量更新已经缓解原有的
热点库存行锁竞争。当前数据库压力更多来自完整抽奖事务和 Outbox 状态逐条更新。

## 结论

- 批量库存同步改造有效：相同并发下 QPS 提升约 `35%～39%`，平均延迟下降约 `26%～28%`。
- 所有档位业务成功率均为 `100%`，没有网络、HTTP 或响应解析错误。
- 并发从 25 增加到 40 后，QPS 不再提高，但平均延迟由 `222ms` 增加到 `359ms`，说明系统已经进入排队区间。
- 当前 `2C4G` 单机环境的性能拐点约为并发 25、`112 QPS`；继续增加并发只会扩大延迟和 Outbox 积压。
- MySQL 是当前主要资源消耗者；Redis、RabbitMQ 和压测程序不是本轮主要瓶颈。
- 下一步应优先验证生产订单表索引，并将 Outbox `completed/fail` 状态改为按分库批量更新，而不是直接继续提高扫描并发。

## 适用范围与限制

- 本结果是单机部署基线，压测程序与被测服务共享同一台 `2C4G` 主机资源。
- 当前接口包含订单、额度、中奖记录、Outbox 和库存同步等完整可靠性链路，不能与旧版纯缓存轻链路直接比较。
- Outbox 数量是数据库累计快照，受到历史测试数据影响；后续测试应记录压测前后增量和积压消化时间。
- 本次没有继续执行并发 60，因为并发 40 已经表现出明显的吞吐平台和延迟增长。
